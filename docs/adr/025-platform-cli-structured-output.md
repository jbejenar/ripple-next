# ADR-025: Platform CLI with Structured JSON Output

| Field        | Value                          |
| ------------ | ------------------------------ |
| **Status**   | Proposed                       |
| **Date**     | 2026-03-03                     |
| **Deciders** | Platform team                  |
| **Relates**  | ADR-018 (AI-first workflow), ADR-024 (Secrets schema) |

## Context

AI coding agents interact with the ripple-next platform through CLI commands.
Currently, agents must call a mix of tools — `sst`, `drizzle-kit`, `pnpm`
scripts, and the AWS CLI — each with different output formats, error handling,
and conventions. This fragmentation is a significant source of agent friction.

Key problems:

1. **Inconsistent output** — `sst deploy` emits human-readable logs, `drizzle-kit`
   outputs coloured terminal text, and AWS CLI returns JSON with AWS-specific
   error codes. Agents cannot reliably parse any of these.
2. **No actionable suggestions** — when a deploy fails because a secret is
   missing, the agent receives a CloudFormation error that requires AWS domain
   knowledge to interpret.
3. **No unified surface** — agents must know which tool to use for which task.
   Database operations require drizzle-kit, deploys require SST, secrets require
   AWS CLI. This cognitive overhead is the opposite of AI-agent-first.

ADR-018 established runbooks and error taxonomy as the foundation for AI-first
workflows. This ADR builds on that by introducing a unified CLI that wraps all
platform operations behind a single, structured interface.

## Decision

Create a `pnpm rip` CLI (`@ripple-next/cli`) that wraps all platform operations
and enforces a universal output contract.

### Command Result Contract

Every command, when invoked with `--json`, returns:

```typescript
interface CommandResult {
  ok: boolean
  code: string              // RPL-XXX-NNN taxonomy code
  message: string           // Human-readable summary
  data: Record<string, any> // Machine-readable payload
  suggestions?: string[]    // CLI commands the agent can run next
  runbook?: string          // Path to relevant runbook
  duration_ms: number       // Execution time
}
```

### Command Surface

| Command                    | Wraps           | Purpose                                 |
| -------------------------- | --------------- | --------------------------------------- |
| `rip secrets list`         | AWS SSM/SM      | List all secrets for a stage            |
| `rip secrets get <KEY>`    | AWS SSM/SM      | Read a secret value                     |
| `rip secrets set <KEY>`    | AWS SSM/SM      | Write a secret                          |
| `rip secrets required`     | Schema          | Show missing secrets                    |
| `rip secrets audit`        | Schema + AWS    | Full compliance report                  |
| `rip env validate`         | Schema          | Validate all env vars for a stage       |
| `rip env diff`             | Schema + AWS    | Compare env between stages              |
| `rip deploy`               | `sst deploy`    | Deploy with pre/post validation         |
| `rip status`               | AWS             | Health of all deployed services         |
| `rip db migrate`           | `drizzle-kit`   | Run migrations with safety checks       |
| `rip db seed`              | Seed scripts    | Seed development data                   |
| `rip deps check`           | Package analysis| Cross-service dependency audit          |

### Error Taxonomy Extension

New error codes in `docs/error-taxonomy.json`:

| Code          | Meaning                                    |
| ------------- | ------------------------------------------ |
| `RPL-SEC-001` | Missing required secret                    |
| `RPL-SEC-002` | Secret value failed format validation      |
| `RPL-SEC-003` | Secret exists in schema but not in store   |
| `RPL-SEC-004` | Structured secret missing required field   |
| `RPL-SEC-005` | Secret rotation overdue                    |
| `RPL-CLI-001` | Unknown command                            |
| `RPL-CLI-002` | Invalid flag combination                   |
| `RPL-CLI-003` | Stage not specified and no default set     |
| `RPL-DEP-001` | Pre-deploy validation failed               |
| `RPL-DEP-002` | Deploy failed — SST error                  |
| `RPL-DEP-003` | Deploy failed — missing required secrets   |
| `RPL-DEP-004` | Post-deploy health check failed            |

### Agent Interaction Example

```
Agent: pnpm rip secrets required --stage production --json
{
  "ok": false,
  "code": "RPL-SEC-003",
  "message": "3 required secrets missing for stage production",
  "data": {
    "missing": [
      {
        "key": "STRIPE_SECRET_KEY",
        "kind": "api-key",
        "canAutoGenerate": false,
        "hasDevDefault": true
      }
    ],
    "present": [ ... ],
    "autoGenerated": ["SESSION_SECRET"]
  },
  "suggestions": [
    "pnpm rip secrets set STRIPE_SECRET_KEY <value> --stage production",
    "pnpm rip secrets set MULESOFT_CLIENT '{\"clientId\":\"...\"}' --stage production"

```

### Relationship to existing pnpm scripts

The existing `pnpm verify`, `pnpm doctor`, `pnpm db:migrate` etc. remain
unchanged. `pnpm rip` wraps these for agent consumption with structured output,
not replace them. Advanced users and human developers can still call the
underlying tools directly.

## Consequences

### Positive

- Agents interact with one tool, one output format, one error taxonomy
- Every failure includes actionable next steps
- Human developers also benefit from the unified CLI
- `--json` makes the CLI composable in scripts and agent tool chains

### Negative

- Maintenance burden — every upstream tool change (SST, drizzle-kit) may
  require CLI wrapper updates
- Indirection layer may obscure underlying errors if not carefully designed

### Neutral

- Does not replace underlying tools — advanced users can still call `sst` or
  `drizzle-kit` directly
- CLI is a dev dependency, not a runtime dependency
