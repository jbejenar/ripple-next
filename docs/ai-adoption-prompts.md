## 1. Greenfield Adoption Prompt

Copy everything below the line and paste it into your AI agent to bootstrap a
new downstream project from scratch.

---

````markdown
## Task: Adopt ripple-next — Greenfield Project

### Project Context

<!-- Fill in ALL fields below before giving this prompt to your AI agent -->

- **Project name (kebab-case):** [my-project]
- **GitHub org:** [my-org]
- **One-line description:** [What this project does and why it exists]
- **App type:** [frontend-only / backend-only / full-stack]
- **Packages needed:** [e.g., @ripple-next/ui, @ripple-next/validation, @ripple-next/auth, @ripple-next/db]
- **Has API endpoints?** [yes / no]
- **Has database?** [yes / no]
- **Target repo path:** [/path/to/repo]

### What You Must Do

Follow the ripple-next golden-path adoption flow end-to-end. The authoritative
references are:

- **Implementation:** `docs/consumer-app-guide.md` in the ripple-next repo
- **Governance:** `docs/downstream-adoption-guide.md` in the ripple-next repo
- **Machine-readable steps:** `docs/runbooks/adopt-ripple-next.json`
- **Available packages:** `docs/platform-capabilities.md`

### Step-by-Step

#### Phase 1: Scaffold

```bash
# From the ripple-next repo root:
pnpm generate:scaffold [TARGET_DIR] --name=[PROJECT_NAME] --org=[ORG] --dry-run
# Review the output, then run without --dry-run:
pnpm generate:scaffold [TARGET_DIR] --name=[PROJECT_NAME] --org=[ORG]
```

This creates ~35+ files: CLAUDE.md, AGENTS.md, agent profiles, CI workflows,
quality gates, documentation templates, fleet governance config.

#### Phase 2: Configure

```bash
cd [TARGET_DIR]
cp .env.example .env
```

- Update `.github/CODEOWNERS` with the team
- Customise `CLAUDE.md` — project-specific constraints (~40 lines max)
- Customise `AGENTS.md` — stack table, repo structure, conventions
- Set up private registry (`.npmrc`) per the Consumer App Guide

#### Phase 3: Install Packages

```bash
pnpm add [PACKAGES_NEEDED]
pnpm add -D @ripple-next/testing   # always needed for conformance tests
```

#### Phase 4: Implement

Follow the Consumer App Guide for the chosen app type ([APP_TYPE]):
- **frontend-only** → Section A (Nuxt app with Ripple UI)
- **backend-only** → Section B (Node service with validation/auth)
- **full-stack** → Section C (pnpm workspace with both)

Key conventions:
- Use the **provider pattern** for all infrastructure (auth, storage, email, queue, events)
- Use **oRPC** for API endpoints (if applicable)
- Use **Drizzle ORM** via `@ripple-next/db` for database access (if applicable)
- Use the **repository pattern** for database queries
- Tests **always** use memory/mock providers — never cloud services
- Lambda is the default compute; use ECS only as an escape hatch

#### Phase 5: Document (7 mandatory categories)

Every downstream repo must produce these 7 documentation artifacts.
See the Downstream Adoption Guide for exact format requirements.

1. **Product Roadmap** → `docs/product-roadmap/README.md`
   - Now/Next/Later timeline with numbered items (e.g., `[PREFIX]-001`)
   - Status, priority, impact, effort, AI-first benefit per item
   - Minimum 3 items describing current work

2. **Architecture Documentation** → `docs/architecture.md`
   - System overview (Mermaid diagram)
   - Stack table (layer, technology, packages consumed, ADRs)
   - Provider pattern usage per environment
   - Compute decisions and deviations from golden path

3. **API Contract Documentation** → `docs/api-contracts.md` (if repo has APIs)
   - Endpoint table, OpenAPI spec location, contract testing approach
   - Run `pnpm generate:openapi` for oRPC repos

4. **Architecture Decision Records** → `docs/adr/`
   - ADR-001 is generated by scaffold
   - Write at least one additional ADR for the most significant decision

5. **Readiness Manifest** → `docs/readiness.json`
   - Replace the scaffold `example` placeholder with real subsystems
   - Include status, maturity, blockers, and test coverage per subsystem

6. **README.md** — project purpose, setup instructions, architecture overview

7. **CLAUDE.md** — verify command, lint rules, key patterns (~40 lines)

#### Phase 6: Verify

```bash
# Conformance scoring (target: >= 70/100)
pnpm conform -- --target=[TARGET_DIR] --json

# Fleet compliance
pnpm check:fleet-drift -- --target=[TARGET_DIR]

# Quality gates
cd [TARGET_DIR] && pnpm verify
```

All three must pass. If conformance score is below 70, review the rubric
output and fix the gaps.

#### Phase 7: Commit and Deploy

- Commit all files to the downstream repo
- Set up CI using the scaffolded GitHub Actions workflows
- Follow the Deployment Guide and Lambda vs ECS decision framework
- Use SST v3 for infrastructure

### Success Criteria

- [ ] Scaffold generated (~35+ files)
- [ ] App runs locally (`pnpm dev`)
- [ ] All tests pass (`pnpm test`)
- [ ] All quality gates pass (`pnpm verify`)
- [ ] Conformance score >= 70 (`pnpm conform`)
- [ ] No fleet drift (`pnpm check:fleet-drift`)
- [ ] All 7 documentation categories completed
- [ ] At least 2 ADRs exist
- [ ] `readiness.json` has real subsystems (not placeholder)
- [ ] Product roadmap has real items (not scaffold template)
````

## 2. Legacy Migration Prompt

Copy everything below the line and paste it into your AI agent to migrate
an existing API or service into ripple-next conventions.

---

````markdown
## Task: Migrate Legacy API to ripple-next

### Project Context

<!-- Fill in ALL fields below before giving this prompt to your AI agent -->

- **Project name (kebab-case):** [my-migrated-api]
- **GitHub org:** [my-org]
- **One-line description:** [What the legacy service does]
- **Legacy technology:** [e.g., Mule 3, .NET, Java Spring, Express, Rails]
- **Legacy API spec location:** [path or URL to OpenAPI/WSDL/RAML spec, or "none"]
- **Number of endpoints to migrate:** [approximate count]
- **Has database?** [yes / no — describe schema if yes]
- **Auth model:** [e.g., API key, OAuth2, session-based, none]
- **Target repo path:** [/path/to/repo]
- **Known edge cases or undocumented behaviour:** [describe any quirks]

### What You Must Do

Follow the ripple-next legacy migration flow end-to-end. The authoritative
references are:

- **Migration procedure:** `docs/runbooks/migrate-legacy-api.json`
- **Adoption governance:** `docs/downstream-adoption-guide.md` (Legacy Migration Path)
- **Implementation:** `docs/consumer-app-guide.md`
- **API conventions:** `docs/api-contracts.md` and ADR-021

### Step-by-Step

#### Phase 1: Analyze

Before writing any code, thoroughly understand the legacy system:

1. **Inventory all endpoints** — method, path, auth model, request/response schemas
2. **Identify data models** — database schemas, entity relationships, data types
3. **Document integration points** — upstream dependencies, downstream consumers, webhooks
4. **Capture edge cases** — undocumented behaviours, coercions, defaults, error formats
5. **Classify each endpoint:**
   - **Migrate** — move to ripple-next with full parity
   - **Deprecate** — remove (no consumers depend on it)
   - **Bridge** — temporary proxy during transition period

Write the inventory to `docs/migration-notes.md` with a table:

| Legacy Endpoint | Method | Path | Classification | Notes |
|----------------|--------|------|---------------|-------|
| ... | ... | ... | migrate/deprecate/bridge | ... |

#### Phase 2: Scaffold

```bash
pnpm generate:scaffold [TARGET_DIR] --name=[PROJECT_NAME] --org=[ORG]
cd [TARGET_DIR]
cp .env.example .env
```

Install packages based on the migration needs.

#### Phase 3: Model

1. **Map legacy data models** to Drizzle ORM schemas
2. **Create Zod validation schemas** for each endpoint's request/response
3. **Map legacy auth** to the OIDC provider pattern
   - `MockAuthProvider` in tests, `OidcAuthProvider` in production
4. Generate database migrations if applicable

#### Phase 4: Contract

1. **Create target OpenAPI spec** matching legacy endpoints
2. For each migrated endpoint, document in `docs/migration-notes.md`:
   - **Current contract (as-is)** — legacy endpoint behaviour
   - **Target contract (to-be)** — ripple-next endpoint behaviour
   - **Breaking changes** — field renames, type changes, removed fields
3. **Generate oRPC routers** matching the target contract
4. Run `pnpm generate:openapi` to produce the spec

#### Phase 5: Implement

1. Implement each migrated endpoint using oRPC + provider pattern + repository pattern
2. Implement **error mapping** — translate legacy error formats to standard responses
3. Use **memory providers** for all tests
4. Document error code mappings in `docs/migration-notes.md`

#### Phase 6: Verify Parity

1. **Write parity tests** comparing legacy and new API responses
2. Cover: success paths, validation failures, error cases, edge cases
3. Verify: status codes, response schemas, field names, null/empty behaviour
4. Run `pnpm test` — all parity tests must pass

#### Phase 7: Document

Complete all 7 mandatory documentation categories (see greenfield prompt above),
plus migration-specific docs:

- `docs/migration-notes.md` — legacy-to-new endpoint mapping with as-is/to-be contracts
- An ADR documenting the migration decision and approach
- Update `readiness.json` with migrated subsystem statuses
- Product roadmap must include legacy system sunset timeline

#### Phase 8: Verify

```bash
pnpm conform -- --target=[TARGET_DIR] --json   # score >= 70
pnpm check:fleet-drift -- --target=[TARGET_DIR]
cd [TARGET_DIR] && pnpm verify
```

### Success Criteria

- [ ] All endpoints classified (migrate/deprecate/bridge)
- [ ] `docs/migration-notes.md` complete with as-is/to-be contracts
- [ ] All migrated endpoints have oRPC implementations
- [ ] Parity tests pass for every migrated endpoint
- [ ] All quality gates pass (`pnpm verify`)
- [ ] Conformance score >= 70
- [ ] All 7 documentation categories completed
- [ ] Migration ADR written
- [ ] Product roadmap includes sunset timeline
````

## 3. Add Feature Prompt (Existing Downstream Repo)

Copy everything below the line and paste it into your AI agent to add a
feature to an existing downstream repo that already follows ripple-next conventions.

---

````markdown
## Task: Add Feature to Downstream Repo

### Feature Context

<!-- Fill in ALL fields below before giving this prompt to your AI agent -->

- **Feature name:** [short descriptive name]
- **Description:** [what the feature does and why it's needed]
- **Affected subsystems:** [e.g., auth, API, UI, database, queue]
- **New API endpoints?** [yes / no — describe if yes]
- **Database changes?** [yes / no — describe if yes]
- **New provider needed?** [yes / no — describe if yes]
- **Roadmap item ID:** [e.g., MY-003, or "new item"]

### What You Must Do

Implement the feature following ripple-next conventions. Before starting:

1. Read `docs/readiness.json` to understand current subsystem status
2. Read `AGENTS.md` for conventions and required validation per change type
3. Read `docs/product-roadmap/README.md` for context on priorities

### Implementation Rules

- **Provider pattern:** Every infrastructure concern uses a provider interface.
  Tests always use memory/mock providers — never cloud services.
- **API endpoints:** Use oRPC with Zod schemas. Run `pnpm generate:endpoint <router> <procedure>`
  to scaffold. Classify visibility (public/internal). Use `protectedProcedure` for auth.
- **Database changes:** Use Drizzle ORM. Generate migrations with `pnpm db:generate`.
  Use the repository pattern for all queries.
- **New providers:** Run `pnpm generate:provider <name> --concern=<concern>`.
  Must pass the conformance suite from `@ripple-next/testing`.
- **Testing:** Write tests for all new code. Meet coverage thresholds
  (Tier 1: 60%, Tier 2: 40%, Tier 3: 20%). Use memory/mock providers.
- **Error handling:** Use typed errors with RPL-xxx taxonomy codes where applicable.

### After Implementation

1. **Update `docs/readiness.json`** if you added or changed a subsystem
2. **Update `docs/product-roadmap/README.md`** — check off completed items or add new ones
3. **Update `AGENTS.md`** if a new subsystem was added
4. **Update `docs/api-contracts.md`** if API endpoints were added/changed
5. **Write or update an ADR** if a significant architecture decision was made
6. **Run `pnpm generate:openapi`** if oRPC routes were added/changed

### Verify

```bash
pnpm verify                    # all quality gates
pnpm check:readiness           # docs not stale
pnpm check:fleet-drift         # no fleet drift (if applicable)
```

### Success Criteria

- [ ] Feature implemented following provider pattern
- [ ] Tests written and passing (meeting coverage thresholds)
- [ ] All quality gates pass (`pnpm verify`)
- [ ] `docs/readiness.json` updated (if subsystem changed)
- [ ] Product roadmap updated
- [ ] API contracts updated (if endpoints changed)
- [ ] No lint errors, no type errors
````

## Using These Prompts

1. **Choose the right prompt** — greenfield for new projects, migration for legacy APIs,
   add-feature for existing repos
2. **Fill in every `[PLACEHOLDER]`** — the more context you provide, the better the agent performs
3. **Paste into your AI agent** (Claude Code, GitHub Copilot, etc.)
4. **Let the agent work** — it has everything it needs to follow the process end-to-end
5. **Verify the output** — always review conformance scores and test results
